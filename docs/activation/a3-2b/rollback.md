# A3.2b Rollback

Owner: Joel / Founder. Trigger on runtime failure, secret exposure, public/active product, nonzero inventory, unauthorized Auth user, Storage policy failure, reconciliation mismatch, checkout/payment exposure or commit mismatch. Roll back Railway to the prior deployment and delete only rows/objects tied to the import execution ID/manifest. Lovable remains unchanged; its 14-day clock does not start in this sprint.

`npm run catalog:rollback:preview:synthetic` validates local output format and scoping logic using a synthetic fixture. It does not query a real database, does not establish readiness, is ineligible as A3.2b execution evidence, and always emits explicit `NON_PRODUCTION_SYNTHETIC` classification fields. The former `catalog:rollback:preview` name remains only as a compatibility alias and emits the same warning.

CI runs `npm run test:catalog-rollback:db` against disposable PostgreSQL with the canonical migrations plus the pending A3.2b migration. It proves that the execution's review, media, and ownership rows are removed before exclusively owned products, that unrelated products and protected commerce rows remain, and that the execution audit row ends as `rolled_back`. A completed execution ID becomes a verified idempotent no-op; an unknown or malformed ID fails closed. The tooling rejects canonical project `wlrfknmrhowldygmvtvn` before any SQL and contains no production rollback adapter.
